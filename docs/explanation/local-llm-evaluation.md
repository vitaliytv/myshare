# Дослідження: локальні LLM (Ollama) для `myshare`

Документ фіксує результати оцінки локальних моделей через Ollama для двох задач
`myshare`: (1) **переклад субтитрів EN→UA** (фіча застосунку) і (2) **агентне
кодування** — запуск скіла `/n-fix` через [`pi`](https://pi.dev). Мета — зрозуміти,
де локальні моделі придатні, де ні, і **чому**. На основі цього згодом
формулюватимемо пропозиції щодо покращення (розділ наприкінці — поки заготовка).

> Це дослідницький знімок, а не специфікація. Цифри відображають конкретне
> залізо, версії та квантизації на момент тесту.

## Стенд

- **Залізо:** Apple **M5**, **16 GB** unified RAM, 10 ядер (4P+6E), macOS 26.5.
- **Runtime:** Ollama 0.24, усі моделі — квантизація **Q4_K_M** (як завантажує Ollama за замовч.).
- **Агентний харнес:** `pi` 0.78 (coding-agent з tools read/bash/edit/write), Ollama підключено
  як OpenAI-сумісний провайдер через `~/.pi/agent/models.json` (`baseUrl http://localhost:11434/v1`),
  `OLLAMA_CONTEXT_LENGTH=32768`.
- **Замір RAM:** поле `size` з `GET /api/ps` (ваги + KV-кеш + буфери поточно завантаженої моделі).

**Вхід:** англійські субтитри одного відео (~2089 символів), однаковий промпт-перекладач,
`temperature 0.2`, `num_ctx 8192`, thinking вимкнено. Метрики: швидкість (tok/s), час

### Висновки (переклад)

- **`gemma3:4b` — оптимум для 16 GB:** 4.4 G RAM (без свопу разом з IDE), ~37 tok/s,
  якість друга після `gemma4:e4b`. Це й закріплено дефолтом у застосунку.
- **`gemma4:e4b` — найкраща якість, але 10.6 G** → впритул на 16 GB, тисне своп.
- **Переклад-спеціалізовані MT-моделі недоступні:** NLLB-200 / MADLAD-400 (seq2seq) Ollama не
  запускає; TowerInstruct / ALMA не покривають українську. Доступні лише **мультимовні LLM**
  (Cohere Aya, Command-R) — і вони **не перевершили** універсальну Gemma для української.
- **Висновок про RAM:** «ефективна» `gemma4:e4b` важча (10.6 G) за справжні 8B-моделі (~6.5 G),
  бо вантажить ~8B сирих ваг + великий KV — саме вона спричиняла гальмування системи.

## Дослідження 2 — агентний `/n-fix` через `pi`

**Задача:** скіл `/n-fix` (виправити структуру проєкту під `.cursor/rules/` так, щоб
`npx @nitra/cursor fix` був чистим). Це багатокроковий агентний цикл: діагностика → читання
правил → правки файлів → реверифікація.

**Методологія:**
- Кожна модель — на **свіжій повній копії** репо (node_modules — APFS-клон, ізольовано).
- Однаковий **детермінований набір порушень** (інжект): доданий заборонений `yarn.lock`;
  доданий заборонений `prettier` у devDependencies (2 правила); прибраний скрипт `lint-image`
  (4 ❌). Baseline копії — 0 ❌, після інжекту — **10 ❌**.
- Виклик: `pi --provider <p> --model <m> --mode json -p "/skill:n-fix"`, watchdog 600 с.
- Метрика успіху: фінальний `npx @nitra/cursor fix` → **скільки ❌ лишилось** (0 = ідеально).

### Локальні моделі

## Аналіз причин (чому локальні провалили `/n-fix`)

Провал — на **двох рівнях**, обидва підтверджені доказами з логів подій.

### Шар A — tool-calling навіть не запустився

- `qwen2.5-coder`, `granite` емітили виклик як **текст** (`{"name":"read","arguments":{…}}` у
  `content`), 0 подій `tool_execution`. `deepseek-coder-v2` — без tools-capability у шаблоні Ollama.
- **Raw-діагностика:** прямий запит до Ollama з tool-схемою повертає голий JSON **і на нативному
  `/api/chat`, і на `/v1`** — 0/6 структурованих викликів навіть на тривіальному промпті без скіла.
- **Корінь:** розсинхрон **template ↔ вивід моделі** в Ollama-пакеті — модель емітить JSON без
  токенів-роздільників `<tool_call>`, яких чекає парсер. Специфічно для моделі: у `gemma4`/`qwen3`
  шаблони збігаються (виклики працювали), у `qwen2.5-coder`/`granite` — ні. Це **стек+квантизація**,
  а не «інтелект»; не залежить ні від `pi`, ні від розміру контексту.

### Шар B — хто викликає, той некомпетентний

- `gemma4:e4b`: кривий `edit` (поламана схема args) + одна правка зробила `package.json`
  невалідним JSON → впали всі перевірки, що його читають → 10→14.
- `gemma4:e2b`: прочитала й передчасно зупинилась. `llama3.1`: усі правки з помилками.
- `qwen3:8b`: пофіксила 2 чисто, але `lint-image` з неправильним вмістом (не прочитала правило),
  сама додала заборонений `oxfmt`, і timeout.
- Бракує **дисципліни сходження**: «помилка → прочитати → скоригувати → повторити → реверифікувати».

### Чому хмарні впорались

- **Надійний tool-calling** (106–116 структурованих викликів — Шару A не існує).
- **Парадокс:** хмарні теж робили багато помилок (gpt-5.5 — 72!), але **самокоригувались і
  сходились до 0**. Різниця не в безпомилковості, а в **циклі відновлення**, якого нема в локальних.

### Наскрізні фактори

1. Tool-use — RLHF-навичка, тонка в малих локальних моделях.
2. **Q4-квантизація** псує структурований вивід (tool-токени, JSON-схеми) сильніше за прозу.
3. Деградація інструкцій на довгому контексті.
4. **Спеціалізація ≠ агентність:** coding-моделі (qwen2.5-coder, deepseek-coder) провалили саме
   **цикл**, а не кодинг.

## Підсумок

| Задача | Режим | Придатність локальних на 16 GB |
|---|---|---|
| Переклад субтитрів EN→UA | один шот, проза | ✅ так — `gemma3:4b` (4.4 G, ~37 tok/s) |
| Агентний `/n-fix` | багатокроковий цикл + tools | ❌ ні — лише хмара (gpt-5.4-mini/5.5) |

Переклад грає на сильних боках LLM; агентний скіл вимагає рівно того, у чому малі квантовані
локальні моделі найслабші (структуровані виклики, точні правки, самоверифікація).
