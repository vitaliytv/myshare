## ADR Вибір `gemma3:4b` як дефолтної моделі для EN→UA перекладу субтитрів

## Context and Problem Statement
Застосунок отримав функцію перекладу англійських субтитрів українською через локальний Ollama. На Mac з 16 GB unified-пам'яті треба було вибрати модель, яка не витісняє інші процеси у своп і дає якісний переклад. Попередній дефолт `gemma4:e4b` (9.6 GB на диску, ~10.6 GB RAM) заповнював swap до 16 GB і гальмував всю систему.

## Considered Options
* `gemma4:e4b` — найкраща якість, але ~10.6 GB RAM → своп на 16 GB
* `gemma3:4b` — ~4.4 GB RAM, якість близька до e4b, 37 tok/s
* `aya-expanse:8b`, `aya:8b`, `command-r7b` (мультимовні, ~6.4–6.8 GB RAM) — гірша якість перекладу, ніж обидві Gemma-моделі
* `qwen3:1.7b`, `qwen3:4b` — reasoning-режим не глушиться навіть із `think:false`, модель не перекладає
* `gemma3:1b` — зациклюється на повторах

## Decision Outcome
Chosen option: "`gemma3:4b`", because на 9-модельному бенчмарку (текст Rick Astley, однаковий промпт, `num_ctx:8192`, M5 / 16 GB) вона показала найкращий баланс RAM / швидкість / якість: 4.4 GB у пам'яті (2.4× менше за `e4b`), 37 tok/s, повний текст за ~14s, природна українська без втрати структурних тегів `♪`.

### Consequences
* Good, because transcript фіксує очікувану користь: після переходу з `gemma4:e4b` swap впав з ~16 GB до ~9 GB і система перестала «стояти» під час перекладу.
* Bad, because `gemma4:e4b` давала дещо натуральніший переклад (порівняно аналізувались виводи обох); `gemma3:4b` зрідка плутає відмінок дієслова. Transcript не містить підтверджених кількісних негативних наслідків.

## More Information
* Константа в `app/src/ollama.js`: `export const DEFAULT_MODEL = 'gemma3:4b'`
* Бенчмарк-скрипт: `/tmp/bench.py`; переклади збережено `/tmp/tr_<model>.txt`; метрики — `/tmp/bench_results.json`
* RAM заміряється через `POST http://localhost:11434/api/ps` після першого запиту з `keep_alive:'5m'`
* `gemma3:4b` не підтримує function-calling у Ollama (`does not support tools`) → непридатна для агентних задач (лише для генерації тексту)

## ADR Таuri HTTP scope — явні порти для localhost Ollama

## Context and Problem Statement
Після додавання Ollama-перекладу кнопка «Перекласти» не відпрацьовувала: `supadata` і `page-meta` фетчились нормально, а `POST http://localhost:11434/api/chat` — ні. Причина — правило Tauri capabilities.

## Considered Options
* Wildcard `http://**` (початковий стан)
* Явні записи `http://localhost:11434/*` і `http://127.0.0.1:11434/*` разом із `http://localhost:*/*`

## Decision Outcome
Chosen option: "явні записи з портом `:11434`", because wildcard `http://**` в urlpattern не матчить URL із явним нестандартним портом; `https://**` працював, оскільки HTTPS-порт дефолтний (прихований). Додано чотири записи в `app/src-tauri/capabilities/default.json`.

### Consequences
* Good, because transcript фіксує очікувану користь: після правки і рестарту dev-застосунку кнопка «Перекласти» запрацювала.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
* Файл: `app/src-tauri/capabilities/default.json`, ключ `permissions[].identifier = "http:default"`, масив `allow`
* Зміни підхоплюються лише після рестарту Tauri-застосунку (capabilities вшиваються в Rust-контекст під час збірки)

## ADR `keep_alive: '5m'` і `num_ctx: 8192` для Ollama-запитів у застосунку

## Context and Problem Statement
Потрібно було визначити стратегію управління ресурсами Ollama-моделі під час перекладу: як довго тримати модель у пам'яті між запитами і яким контекстним вікном обмежити KV-кеш.

## Considered Options
* `keep_alive: 0` — негайне вивантаження після перекладу (зменшує слід у RAM одразу)
* `keep_alive: '5m'` — тримати 5 хв (поведінка між чанками і при повторних перекладах)
* Без явного `num_ctx` — модель сама обирає (дефолт gemma3 = 32k+ → великий KV-кеш)
* `num_ctx: 8192` — обмеження KV-кешу

## Decision Outcome
Chosen option: "`keep_alive: '5m'` + `num_ctx: 8192`", because перший варіант (`keep_alive: 0`) вивантажував модель після кожного чанку, що спричинило б повторне завантаження 4.4 GB між чанками одного перекладу. Користувач явно підтвердив повернення до 5 хв після того, як ця опція була тимчасово виставлена на 0. `num_ctx: 8192` достатній для чанку ≤3500 символів (~1200 токенів) + перекладу — лишає запас і скорочує KV-кеш.

### Consequences
* Good, because transcript фіксує очікувану користь: модель залишається завантаженою між чанками одного перекладу і при повторних тапах «Перекласти» протягом 5 хв — без повторного холодного старту.
* Bad, because 4.4 GB у RAM лишаються ще 5 хв після завершення перекладу. Neutral, because на 16 GB це прийнятно порівняно з ситуацією до вибору легшої моделі.

## More Information
* `app/src/ollama.js`: параметри `keep_alive` і `num_ctx` у тілі `POST /api/chat`
* Чанкер: `chunkText` (≤3500 символів, по межах абзаців), `app/src/ollama.js`

## ADR Підключення Ollama до pi через OpenAI-сумісний endpoint і `~/.pi/agent/models.json`

## Context and Problem Statement
Для тестування локальних LLM в агентних задачах (скіл `n-fix`) потрібно було підключити Ollama до pi coding-agent CLI замість хмарних провайдерів.

## Considered Options
* Змінні середовища `OPENAI_BASE_URL` + `OPENAI_API_KEY=ollama` при запуску `pi --provider openai` — pi ігнорує `OPENAI_BASE_URL` і звертається до справжнього api.openai.com (401)
* Файл `~/.pi/agent/models.json` з кастомним провайдером `api: "openai-completions"` і `baseUrl: "http://localhost:11434/v1"` — підтримується офіційно, знайдено в `pi/docs/models.md`

## Decision Outcome
Chosen option: "`~/.pi/agent/models.json` з кастомним провайдером `ollama`", because pi ігнорує env-змінні для base URL і надає офіційний механізм кастомних провайдерів через цей JSON-файл. Ollama реалізує OpenAI-сумісний `/v1` endpoint, тож `api: "openai-completions"` достатній.

### Consequences
* Good, because transcript фіксує очікувану користь: `pi --list-models ollama` почав показувати обидві моделі, tool-calling пройшов перший тест (`qwen2.5-coder:7b` викликав `bash echo TOOLS_OK`).
* Bad, because tool-call emitting виявився ненадійним: модель іноді вивводила `{"name":"bash",...}` як текст у markdown-блоці замість структурованого `tool_call` — pi не виконував такі виклики. Neutral, because це обмеження Ollama-shim, а не самого `models.json`.

## More Information
* Файл: `~/.pi/agent/models.json`; провайдер `ollama`, `baseUrl: "http://localhost:11434/v1"`, `apiKey: "ollama"`, `compat.supportsReasoningEffort: false`, `compat.supportsDeveloperRole: false`
* Документація pi: `/opt/homebrew/Cellar/pi-coding-agent/0.78.0/.../docs/models.md`
* Контекст встановлено 32 768 токенів (для агентних задач)

## ADR Локальні моделі через Ollama непридатні для агентного скілу `n-fix` у pi

## Context and Problem Statement
Тест 6 локальних моделей (qwen2.5-coder:7b, granite3.3:8b, llama3.1:8b, qwen3:8b, deepseek-coder-v2:16b, gemma4:e4b) через реальний `/skill:n-fix` у pi з фіксованим набором порушень (yarn.lock, prettier у devDeps, відсутній скрипт lint-image) — щоб визначити, яка з них підходить для агентного кодування замість хмарних Claude-моделей.

## Considered Options
* qwen2.5-coder:7b — coding-спеціалізована, tool-capable
* granite3.3:8b — IBM tool-орієнтована
* llama3.1:8b — Meta, tools підтримує
* qwen3:8b — reasoning-модель, tools підтримує
* deepseek-coder-v2:16b — MoE coder
* gemma4:e4b — найкраща якість для перекладу

## Decision Outcome
Chosen option: "жодна не підходить для `n-fix` через pi/Ollama у поточній конфігурації", because:
- **qwen2.5-coder:7b, granite3.3:8b**: `tools=0` — emitять tool-виклики як текст (`{"name":"bash",...}` у markdown), pi не виконує їх; after=10/10.
- **deepseek-coder-v2:16b**: `does not support tools` у Ollama; after=10.
- **llama3.1:8b**: tools=3, але 9 помилок, after=10 — цикл ламається.
- **gemma4:e4b**: 6 викликів, 9 помилок, after=14 (погіршила стан — додала нові порушення).
- **qwen3:8b**: єдина з прогресом (after=7 з 10), але 474 с і 5 помилок — занадто повільна й ненадійна.

### Consequences
* Good, because тест чітко виявив вузьке місце: проблема не в здатності моделей кодувати, а в ненадійному emitting структурованих `tool_calls` через Ollama OpenAI-compat shim — знання застосовне до будь-яких агентних задач через цей стек.
* Bad, because локальні LLM через Ollama + pi не замінюють хмарні Claude-моделі для агентних задач типу `n-fix` при поточному стеку. Neutral, because `qwen2.5-coder:7b` може поводитись краще з нативним API (Cursor, Continue.dev з FIM) — transcript не містить підтвердження цього.

## More Information
* Раннер: `/tmp/nfix_run2.sh`; батч: `/tmp/nfix_batch2.sh`
* Стенд: rsync-копія репо в `/tmp/nf2_<model>/.worktrees/run-fix/` (шлях із `.worktrees/` обходить worktree-preflight скіла)
* Порушення-інжектор: `/tmp/inject_violations.sh` (yarn.lock, prettier у devDeps, видалений lint-image)
* Верифікація: `npx @nitra/cursor fix 2>&1 | grep -c '❌'`
* `gemma3:4b` не брала участі в тесті: Ollama підтверджує `does not support tools` для цієї моделі
