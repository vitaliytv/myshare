## ADR Вибір `gemma3:4b` дефолтною моделлю для перекладу субтитрів EN→UA

## Context and Problem Statement
Фіча перекладу субтитрів у `myshare` потребує локальної LLM через Ollama. На машині M5/16 GB потрібно вибрати одну модель-дефолт, яка дає придатний переклад українською без свопування.

## Considered Options
* `gemma3:4b` — 3.3 GB диску, RAM 4.4 G, 37.5 tok/s
* `gemma4:e4b` — 9.6 GB диску, RAM 10.6 G, 29.1 tok/s
* Мультимовні LLM `aya-expanse:8b`, `aya:8b`, `command-r7b` (~5–6.8 G RAM)
* MT-спеціалізовані NLLB-200 / MADLAD-400 (seq2seq)

## Decision Outcome
Chosen option: "`gemma3:4b`", because benchmark на 12 моделях показав: якість перекладу — друга після `gemma4:e4b` (явних семантичних помилок нема, лише дрібна плутанина особи дієслова), RAM 4.4 G не тисне своп разом з IDE, а повний текст перекладається за ~14 с. `gemma4:e4b` (10.6 G) систематично тиснула своп і сповільнювала систему; мультимовні (Aya, Command-R) мали гіршу якість при більшому розмірі; MT-seq2seq моделі Ollama взагалі не запускає.

### Consequences
* Good, because transcript фіксує очікувану користь: переклад завершується без свопу (~14 с, 4.4 G RAM), якість перевірена на реальних субтитрах.
* Bad, because `gemma4:e4b` дає помітно природніший переклад — якщо якість стане пріоритетом і пам'ять буде вільна, дефолт доведеться переключати вручну.

## More Information
Файл з дефолтом: `DEFAULT_MODEL = 'gemma3:4b'`. Бенчмарк: `/tmp/bench.py`, результати `/tmp/bench_results.json`. Замір RAM через `GET /api/ps` (ваги + KV-кеш + буфери). MT-спеціалізовані NLLB-200/MADLAD-400 не підтримуються llama.cpp — encoder-decoder архітектура. TowerInstruct/ALMA відсутні в реєстрі Ollama й не покривають українську.

## ADR Підключення Ollama до `pi` через OpenAI-сумісний endpoint

## Context and Problem Statement
Для бенчмарку агентних скілів через `pi` (coding-agent CLI) потрібно підключити локальні Ollama-моделі. `pi` не має вбудованого ollama-провайдера, але підтримує кастомні провайдери через `~/.pi/agent/models.json`.

## Considered Options
* Кастомний провайдер у `~/.pi/agent/models.json` з `baseUrl http://localhost:11434/v1` (OpenAI-compat)
* Змінна середовища `OPENAI_BASE_URL` — виявилась проігнорованою (`pi` пішов на реальний OpenAI)
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "Кастомний провайдер у `~/.pi/agent/models.json`", because `OPENAI_BASE_URL` `pi` ігнорує (підтверджено тестом — отримали 401 від OpenAI); `models.json` дає повний контроль (baseUrl, api: `openai-completions`, apiKey: `ollama`, context 32k).

### Consequences
* Good, because transcript фіксує очікувану користь: `pi --list-models ollama` бачить усі моделі, `PONG`-тест проходить.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Файл: `~/.pi/agent/models.json`. Налаштування: `baseUrl: "http://localhost:11434/v1"`, `api: "openai-completions"`, `apiKey: "ollama"`, контекст 32 768 токенів. `OLLAMA_CONTEXT_LENGTH=32768` передається при старті сервера. Документація: `pi` docs/models.md → розділ «Custom Models».

## ADR Локальні моделі не придатні для агентного `/n-fix`; хмарні — єдиний варіант

## Context and Problem Statement
Потрібно з'ясувати, чи можна запускати скіл `/n-fix` (виправлення структури проєкту під `.cursor/rules/`) через `pi` на локальних Ollama-моделях замість хмарних — задля економії та приватності.

## Considered Options
* Локальні моделі (qwen2.5-coder:7b, qwen3:8b, gemma4:e4b, gemma4:e2b, llama3.1:8b, granite3.3:8b, deepseek-coder-v2:16b)
* Хмарні через `openai-codex` ChatGPT OAuth (gpt-5.4-mini, gpt-5.5, gpt-5.3-codex)

## Decision Outcome
Chosen option: "Хмарні моделі (gpt-5.4-mini / gpt-5.5)", because benchmark з ідентичним набором з 10 ❌ показав: жодна локальна модель не досягла 0 ❌ (найкращий результат qwen3:8b — 10→7, з timeout), тоді як gpt-5.4-mini і gpt-5.5 досягли 10→0. Причини провалу локальних — два незалежні шари: (A) qwen2.5-coder/granite емітують tool-виклики текстом (розсинхрон template ↔ вивід в Ollama), deepseek без tool-support; (B) ті, що викликають (gemma4/qwen3), не мають циклу відновлення: gemma4:e4b зламала репо (10→14), qwen3:8b не сходиться.

### Consequences
* Good, because transcript фіксує очікувану користь: хмарні надійно досягають 0 ❌ (навіть попри 20–72 помилки на шляху — завдяки циклу самокорекції).
* Bad, because хмарні витрачають токени підписки ChatGPT і потребують мережі; gpt-5.3-codex взагалі недоступна на ChatGPT-акаунті (потрібен API-ключ).

## More Information
Стенд: `/tmp/nfix_run2.sh` (свіжа rsync-копія репо на кожну модель, однаковий інжект порушень: `yarn.lock`, `prettier` у devDeps, відсутній скрипт `lint-image`). Watchdog 600 с. Логи подій: `/tmp/events2_<model>.jsonl`. ## ADR Детермінований autofix у `@nitra/cursor` як наступний крок замість LLM-скаффолда

## Context and Problem Statement
Після того як агентний n-fix провалився на локальних моделях, а хмарні (gpt-5.5: 116 викликів, 72 помилки, 600 с) витратили надмірно токенів для тривіальних виправлень, постало питання: де правильне місце для логіки фіксу.

## Considered Options
* Скаффолд у `pi`-розширенні або n-fix скілі (verify-gated retry loop, tool-call repair shim, safe structured edit)
* Детермінований `--apply` режим в оркестраторі `@nitra/cursor fix`
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "Детермінований `--apply` режим в `@nitra/cursor fix`", because `@nitra/cursor fix` v3.13 — діагностика-тільки, але кожне rego-правило вже читає `template/*.json`, де лежить **правильна відповідь** (точний вміст скрипту `lint-image`, дозволені deps тощо). Фіксер просто читає той самий темплейт і записує у target — нуль токенів, детерміністично, надійно. Скаффолд у харнесі лікував би симптоми, але кожна правка все одно лишалась за моделлю.

### Consequences
* Good, because transcript фіксує очікувану користь: 95%+ порушень — детерміновані; `npx @nitra/cursor fix --apply` → 0 ❌ за 2–3 с без моделі. Покращення поширюється й на хмарні моделі (їм теж не потрібно «вгадувати»).
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Пропонована структура: поряд із кожним `policy/` додати `fix/` директорію з фіксером (`fix/package_json.ts`, `fix/index.ts`). CLI-прапор `--apply` (або `--fix`): для кожного ❌ шукає парний фіксер, запускає, реверифікує. `@nitra/cursor` v3.13, правила у `node_modules/@nitra/cursor/rules/`. Rego-правила читають `template/package.json.contains.json` / `template/package.json.deny.json`. **Статус:** ідея зафіксована в `docs/explanation/local-llm-evaluation.md`, реалізація — наступний крок.
