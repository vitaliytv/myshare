## ADR Переклад субтитрів EN→UA через локальний Ollama без нової Rust-команди

## Context and Problem Statement
У myshare-додатку потрібно перекладати англійські субтитри українською. Ollama вже встановлений локально; питання — через який канал відправляти HTTP-запити до нього з Tauri-застосунку: через новий Rust-бекенд або через вже наявний HTTP-плагін на стороні JS.

## Considered Options
* Використати `tauri-plugin-http` зі сторони JS (`app/src/ollama.js`) — без нової Rust-команди
* Додати нову Tauri-команду в Rust-бекенд для спілкування з Ollama

## Decision Outcome
Chosen option: "Використати `tauri-plugin-http` зі сторони JS", because Ollama — локальний сервіс на `localhost:11434`; `tauri-plugin-http` вже підключений і дозволяє HTTP-запити; новий Rust-шар не додає цінності, лише складності.

### Consequences
* Good, because transcript фіксує очікувану користь: реалізація `app/src/ollama.js` обійшлась без торкання Rust-коду та Cargo.toml.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
- `app/src/ollama.js` — `POST http://localhost:11434/api/chat` (`stream:false`, `temperature:0.2`)
- `app/src/ollama.js` — `GET http://localhost:11434/api/tags` для `resolveModel`
- `app/src-tauri/capabilities/default.json` — потрібно явно додати `http://localhost:11434/*` і `http://127.0.0.1:11434/*` (патерн `http://**` не матчить URL із явним нестандартним портом)

## ADR Tauri HTTP capabilities: явний запис localhost:11434

## Context and Problem Statement
Після інтеграції Ollama-перекладу HTTP-запити до `http://localhost:11434` блокувались Tauri-runtime, хоча в `capabilities/default.json` стояв дозвіл `http://**`. supadata (HTTPS) і page-meta (HTTPS) при цьому працювали без проблем.

## Considered Options
* Залишити `http://**` і вважати, що wildcard покриває будь-який порт
* Додати явні записи `http://localhost:11434/*` і `http://127.0.0.1:11434/*` поряд із `http://**`

## Decision Outcome
Chosen option: "Додати явні записи", because `http://**` в Tauri URL-pattern не матчить URL із явним нестандартним портом (тоді як HTTPS-URL з дефолтним портом 443 матчить — звідси асиметрія). Явний запис усуває залежність від поведінки wildcard-парсера.

### Consequences
* Good, because transcript фіксує очікувану користь: після додавання явних записів і перезапуску dev-застосунку кнопка «Перекласти» запрацювала.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
- Файл: `app/src-tauri/capabilities/default.json`
- Додані записи: `{ "url": "http://localhost:11434/*" }` і `{ "url": "http://127.0.0.1:11434/*" }`
- Зміна набирає чинності лише після перезапуску застосунку (capabilities вшиваються під час Rust-збірки)

## ADR Вибір gemma3:4b як дефолтної моделі EN→UA перекладу на 16 GB RAM

## Context and Problem Statement
Початковий дефолт `gemma4:e4b` (9.6 GB на диску, ~10.6 GB у RAM) спричиняв критичне гальмування системи: на Mac M5 з 16 GB unified-RAM ваги моделі не вміщались поряд з IDE+build-оточенням, swap досягав 16 GB і диск «молотив». Потрібно обрати дефолтну модель, що забезпечує прийнятну якість перекладу без свопу.

## Considered Options
* `gemma4:e4b` — найвища якість у тесті (28 tok/s, 10.6 G RAM)
* `gemma3:4b` — 37 tok/s, 4.4 G RAM
* `aya-expanse:8b` (Cohere, 6.7 G RAM), `aya:8b` (6.4 G), `command-r7b` (6.8 G) — multilingual-спеціалізовані
* `qwen3:4b`, `qwen3:1.7b` — reasoning-моделі (think не глушиться через Ollama, не перекладають)
* `gemma3:1b`, `qwen3:1.7b` — зациклюються на повторах

## Decision Outcome
Chosen option: "`gemma3:4b`", because це єдина модель, яка одночасно: влазить у RAM без свопу (4.4 G → ~11 G вільних при dev-оточенні), має задовільну якість перекладу (природна українська, повний текст), завершує переклад без зациклення (~14s) і вдвічі швидша за gemma4:e4b.

### Consequences
* Good, because transcript фіксує очікувану користь: swap впав з 16 GB до ~3.8 GB, живий переклад підтвердив коректний результат (`gemma3:4b`, curl-тест).
* Bad, because якість дещо нижча за `gemma4:e4b`: плутає особу дієслова у деяких рядках.

## More Information
- Константа: `DEFAULT_MODEL = 'gemma3:4b'` у `app/src/ollama.js`
- Бенчмарк: 9 моделей на однаковому тексті (Rick Astley, 2089 символів, `num_ctx 8192`, `temperature 0.2`)
- Вивантаження `aya:8b`, `aya-expanse:8b`, `command-r7b`, `gemma4:e4b` та інших після тесту
- Спеціалізовані MT-моделі (NLLB-200, MADLAD-400) не підтримуються llama.cpp і відсутні в реєстрі ollama

## ADR keep_alive: '5m' замість негайного вивантаження після перекладу

## Context and Problem Statement
Першу версію оптимізації для зниження тиску на RAM реалізовано з `keep_alive: 0` — модель вивантажувалась одразу після перекладу. Згодом користувач попросив повернути 5-хвилинне вікно.

## Considered Options
* `keep_alive: 0` — негайне вивантаження після перекладу (звільняє ~4.4 G RAM одразу)
* `keep_alive: '5m'` — модель лишається 5 хв (повторні переклади і завершення чанків без перезавантаження)

## Decision Outcome
Chosen option: "`keep_alive: '5m'`", because модель не перевантажується між чанками одного перекладу та при повторних запусках протягом 5 хв; `gemma3:4b` займає лише 4.4 G, що не критично на 16 GB за умови закритого браузера/другого редактора.

### Consequences
* Good, because transcript фіксує очікувану користь: перший завантажений тест після налаштування `keep_alive:'5m'` показав правильну позначку `UNTIL 4 min` у `ollama ps`.
* Bad, because 4.4 G залишаються в RAM 5 хв після перекладу — при запуску кількох важких процесів одночасно це може спровокувати невеликий тиск.

## More Information
- `app/src/ollama.js`: поле `keep_alive` у тілі `POST /api/chat`
- `num_ctx: 8192` збережено (обмежує KV-кеш; чанк ≤3500 символів ≈ ~1200 токенів влазить із запасом)
- `unloadModel` (`POST /api/generate`, `keep_alive:0`) — функція видалена разом із цим рішенням

## ADR Gemma-родина несумісна з агентним tool-calling в Ollama

## Context and Problem Statement
Під час підготовки бенчмарку локальних моделей для агентної задачі n-fix (через pi.dev) потрібно було визначити, які моделі взагалі здатні виконувати tool-calling (bash/edit/write) в Ollama-shim.

## Considered Options
* Включити `gemma3:4b` і `gemma4:e4b` до агентного тесту
* Виключити всю Gemma-родину

## Decision Outcome
Chosen option: "Виключити всю Gemma-родину", because Ollama при спробі завантажити gemma3:4b з tool-calling повернув `400: does not support tools`; gemma4:e4b технічно не заблокована, але в тесті збільшила кількість ❌ з 10 до 14 (погіршила стан репо), що робить її непридатною для агентних задач.

### Consequences
* Good, because transcript фіксує очікувану користь: чітке розмежування ролей — Gemma підходить для генерації тексту (переклад), але не для агентного кодування.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
- Перевірено командою: `pi --provider ollama --model gemma3:4b --no-session -p "Run bash: echo TOOLS_OK"` → `400 does not support tools`
- `gemma4:e4b` у батчі n-fix: `before=10, after=14, tools=6, err=9, time=168s`
- Tool-capable моделі з Ollama у тесті: `qwen2.5-coder:7b`, `qwen3:8b`, `llama3.1:8b`, `granite3.3:8b`
- `deepseek-coder-v2:16b` також `does not support tools` в Ollama

## ADR pi.dev підключення до Ollama через ~/.pi/agent/models.json (OpenAI-compat)

## Context and Problem Statement
pi.dev (v0.78.0) потрібно налаштувати для роботи з локальними моделями через Ollama, щоб запускати агентні скіли (n-fix) з різними моделями без хмарних провайдерів.

## Considered Options
* Задавати `OPENAI_BASE_URL` і `OPENAI_API_KEY=ollama` через env
* Використовувати `~/.pi/agent/models.json` — офіційний механізм кастомних провайдерів

## Decision Outcome
Chosen option: "`~/.pi/agent/models.json`", because pi ігнорує `OPENAI_BASE_URL` і напряму звертається до реального OpenAI (401); офіційна документація `docs/models.md` описує `models.json` як єдиний підтримуваний шлях для кастомних провайдерів (Ollama, vLLM, LM Studio).

### Consequences
* Good, because transcript фіксує очікувану користь: після запису `models.json` команда `pi --list-models` показала обидві Ollama-моделі; `qwen2.5-coder:7b` успішно викликала bash через pi.
* Bad, because схема `cost` у `models.json` вимагає всі 4 поля (`input`, `output`, `cacheRead`, `cacheWrite`) — часткове заповнення дає `Invalid models.json schema` і блокує завантаження.

## More Information
- Файл: `~/.pi/agent/models.json`
- Провайдер: `api: "openai-completions"`, `baseUrl: "http://localhost:11434/v1"`, `apiKey: "ollama"`
- `compat.supportsToolStreaming: false` — обов'язково, інакше streaming tool-calls ламає парсинг
- Документація: `/opt/homebrew/Cellar/pi-coding-agent/0.78.0/.../docs/models.md` (розділ «Custom Models»)
- Виявлена проблема: `qwen2.5-coder:7b` і `granite3.3:8b` емітують tool-виклики як markdown-текст (`{"name":"bash",...}`), а не структуровані `tool_call` → pi не виконує їх → агентний цикл завершується без дій
