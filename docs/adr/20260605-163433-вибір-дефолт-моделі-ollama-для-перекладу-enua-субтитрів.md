## ADR Вибір дефолт-моделі Ollama для перекладу EN→UA субтитрів

## Context and Problem Statement

Застосунок `myshare` перекладає EN-субтитри YouTube локально через Ollama. Дефолтна модель `gemma4:e4b` займала ~10.6 GB RAM на Mac M5/16 GB, що спричиняло своп-трешинг (`swap used = 16 GB`, 1.6M pageouts) і гальмувало всю систему під час перекладу.

## Considered Options

* `gemma4:e4b` (9.6 GB диск, ~10.6 GB RAM) — попередній дефолт
* `gemma3:4b` (3.3 GB диск, ~4.4 GB RAM)
* `aya-expanse:8b` (~6.7 GB RAM), `aya:8b` (~6.4 GB RAM), `command-r7b` (~6.8 GB RAM) — мультимовні спеціалізовані
* `gemma3:1b`, `qwen3:1.7b`, `ministral-3:3b`, `qwen3:4b`, `mistral:7b`, `ministral-3:8b` — протестовані, відкинуті

## Decision Outcome

Chosen option: "`gemma3:4b`", because бенчмарк 9 моделей на однаковому тексті (Rick Astley EN-субтитри, 2089 символів, M5/16 GB) показав: `gemma3:4b` займає 4.4 GB RAM (100% GPU, без свопу), генерує 37.5 tok/s, і посідає другу позицію за якістю перекладу після `gemma4:e4b`; спеціалізовані мультимовні моделі (Aya, Command-R) важчі на 2 GB, але якість гірша. RAM — ключовий критерій: 4.4 GB лишає headroom для dev-оточення (Zed + VS Code + vite + cargo ≈ 7–8 GB).

### Consequences

* Good, because transcript фіксує очікувану користь: усі переклади у тесті завершились без свопу; RAM після перекладу звільняється через `keep_alive: '5m'` (після 5 хв Ollama вивантажує модель автоматично).
* Bad, because `gemma4:e4b` давала вищу якість (найприродніша українська у тесті); `gemma3:4b` допускає дрібні помилки відмінювання. Transcript фіксує це як свідому компромісну жертву.

## More Information

Зміни в `app/src/ollama.js`:
- `DEFAULT_MODEL` → `'gemma3:4b'` (було `'gemma4:e4b'`).
- `options.num_ctx: 8192` — обмежує KV-кеш; чанк ≤3500 символів ≈ 1200 токенів вміщується з запасом.
- `keep_alive: '5m'` — явне значення замість дефолту (модель тримається між чанками того самого перекладу й повторними запитами впродовж 5 хв).
- Функція `unloadModel` (з тимчасовим `keep_alive: 0`) — видалена: перезавантаження 4.4 GB перед кожним чанком знищувало перевагу меншої моделі.

Замір RAM через `GET http://localhost:11434/api/ps` (поле `size` / `size_vram`). Тест запускався на свіжому `ollama serve` (попередній сервер мав Metal-crash `MTLLibraryErrorDomain Code=3` після кількох годин роботи).

## ADR Локальні LLM не придатні для агентних скілів через pi.dev

## Context and Problem Statement

Проєкт `myshare` виконує агентні скіли (`/n-fix`, `/n-coverage-fix`) через pi.dev CLI. Постало питання, чи можна запускати ці скіли через локальні моделі Ollama замість хмарних (OpenAI Codex, Anthropic), щоб знизити витрати та забезпечити роботу офлайн.

## Considered Options

* Локальні моделі через Ollama + pi.dev (`~/.pi/agent/models.json`, `api: openai-completions`, `baseUrl: http://localhost:11434/v1`)
* Хмарні моделі (OpenAI Codex gpt-5.x, Anthropic) — поточний варіант

## Decision Outcome

Chosen option: "Хмарні моделі", because тестування 7 локальних моделей через реальний `/skill:n-fix` у pi.dev показало: жодна не завершила агентний цикл успішно. Три окремі причини провалу: (1) tool-calling через Ollama `/v1` shim ненадійний — `qwen2.5-coder:7b` і `granite3.3:8b` емітують виклики текстом у полі `content` замість структурованого `tool_calls`, pi.dev не виконує їх; (2) Gemma-родина й `deepseek-coder-v2:16b` в Ollama повертають `does not support tools`; (3) `qwen3:8b` — єдина з реальним прогресом (10→7 ❌), але вклалась у timeout 600с, внесла нове порушення (`oxfmt` у devDependencies).

### Consequences

* Good, because transcript фіксує очікувану користь: хмарні моделі надійно виконують агентний цикл завдяки reinforcement на `tool_calls`-форматі та точному `edit` за специфікацією pi.dev.
* Bad, because `gemma4:e4b` активно зламала репо (10→14 ❌): збила JSON у `package.json` через неточний `oldText` в `edit`-виклику → усі подальші `npx @nitra/cursor fix`-перевірки (lint, scripts, GA, Rust, Vue) падали. Тобто неправильна локальна модель не просто «не допомагає» — вона деструктивна.

## More Information

Тестовий стенд: rsync-копія репо в `/tmp/nf2_<model>/.worktrees/run-fix/` (шлях містить `.worktrees/` — worktree-preflight скіла не плодить вкладених worktree); інжект фіксованих порушень (`yarn.lock`, `prettier` у devDeps, відсутній скрипт `lint-image`); `pi --provider ollama --model <model> /skill:n-fix --mode json`; watchdog 600с (`kill -9 $PIPID`); верифікація `npx @nitra/cursor fix`.
