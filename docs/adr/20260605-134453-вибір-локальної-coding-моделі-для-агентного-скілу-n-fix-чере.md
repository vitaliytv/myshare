## ADR Вибір локальної coding-моделі для агентного скілу n-fix через pi.dev

## Context and Problem Statement
Для запуску агентних скілів (зокрема `n-fix`) через pi.dev CLI з локальним Ollama потрібна модель, що підтримує tool-calling, влізає в 16 GB unified-пам'яті M5 без свопу та виконує агентний цикл (read/bash/edit/write) за прийнятний час.

## Considered Options
* qwen2.5-coder:7b (coding-спеціалізована, 8.0 G RAM, 49s, tool-calling ✅)
* qwen3:8b (загального призначення + thinking, 7.2 G RAM, 44s, tool-calling ✅)
* granite3.3:8b (IBM, 6.8 G RAM, 48s, tool-calling ✅)
* llama3.1:8b (5.7 G RAM, 56s, tool-calling ✅)
* deepseek-coder-v2:16b (MoE, 12.2 G RAM, 99s)
* qwen3:30b-a3b (MoE, 14.1 G RAM, ~600s, межа свопу)
* gemma3:4b та gemma4:e4b — виявились непридатними: Ollama-версії gemma не підтримують tools (`does not support tools`)

## Decision Outcome
Chosen option: **"qwen2.5-coder:7b"**, because coding-спеціалізована, підтримує tool-calling, виконує повний агентний цикл n-fix (12 порушень → 0 за 49s), 8.0 G RAM — без ризику свопу на 16 GB; `qwen3:8b` — рівнозначна альтернатива (44s, 7.2G).

### Consequences
* Good, because transcript фіксує: всі 12 інжектованих порушень виправлено, RAM в безпечній зоні, tool-calling стабільний.
* Bad, because на складніших завданнях 7B може поступитись якістю більшим моделям; deepseek-16b показав кращу точність коду, але 12.2 G RAM — ризик свопу на 16 GB.

## More Information
- Конфіг: `~/.pi/agent/models.json` — Ollama як OpenAI-сумісний провайдер (`baseUrl: http://localhost:11434/v1`, `api: openai-completions`, `num_ctx: 32768`)
- gemma3:4b/gemma4:e4b: `does not support tools` — непридатні для агентних скілів, лише для генерації тексту
- Тест: 7 моделей × rsync-копія репо + інжект yarn.lock/prettier/lint-image → `pi --provider ollama --model … --skill n-fix-test --mode json`
