## ADR Вибір дефолтної моделі Ollama для перекладу субтитрів

## Context and Problem Statement
У Tauri-застосунку `myshare` реалізовано функцію EN→UA перекладу субтитрів через локальний Ollama. Постало питання: яку модель обрати дефолтною з урахуванням 16 GB unified RAM на Apple M5 (де GPU і CPU ділять спільну пам'ять) та необхідності уникнути своп-трешингу під час перекладу.

## Considered Options
* `gemma4:e4b` (9.6 GB на диску, ~10.6 GB у RAM)
* `gemma3:4b` (3.3 GB на диску, ~4.4 GB у RAM)
* `aya-expanse:8b`, `aya:8b`, `command-r7b` (мультимовні спеціалізовані, ~6.4–6.8 GB у RAM)

## Decision Outcome
Chosen option: "`gemma3:4b`", because модель важить ~4.4 GB у RAM (проти ~10.6 GB у `gemma4:e4b`), генерує 37–38 tok/s і видає природний переклад без свопу на 16 GB. Тест на реальних субтитрах Rick Astley підтвердив якість, близьку до `gemma4:e4b`, за вдвічі меншого споживання пам'яті; мультимовні спеціалізовані (Aya, Command-R) показали гіршу якість за вищого RAM-footprint.

### Consequences
* Good, because transcript фіксує очікувану користь: переклад повного тексту (~400 токенів виводу) за ~14 с без своп-трешингу; swap впав з 16 GB до ~3–5 GB після переходу.
* Bad, because `gemma3:4b` не підтримує tool-calling в Ollama (`does not support tools`) — непридатна для агентних задач (n-fix, автокомпліт у редакторі). Це усвідомлене обмеження: модель лишається тільки для генерації тексту.

## More Information
- `DEFAULT_MODEL = 'gemma3:4b'` — константа в `app/src/ollama.js`
- `keep_alive: '5m'` у тілі запиту `POST /api/chat` — модель тримається в RAM 5 хв після перекладу, щоб повторні переклади не перевантажували її щоразу
- `num_ctx: 8192` обмежує KV-кеш (дефолтний контекст gemma4 — 131 072 токенів)
- `resolveModel` в `ollama.js` — підставний механізм: якщо дефолту нема, бере першу доступну модель через `GET /api/tags`
- Тест проводився на субтитрах відео (Rick Astley, 2089 символів) через `POST /api/chat` із `stream:false`, `temperature:0.2`
- Benchmark-скрипт `/tmp/bench.py` зберігав переклади в `/tmp/tr_<model>.txt` і метрики в `/tmp/bench_results.json`

## ADR Налаштування Tauri HTTP scope для Ollama localhost

## Context and Problem Statement
Tauri-застосунок не міг відправляти HTTP-запити до локального Ollama-сервера (`http://localhost:11434`). Функція перекладу повертала мережеву помилку, хоча Ollama-сервер був запущений.

## Considered Options
* Додати `http://localhost:11434/*` і `http://127.0.0.1:11434/*` в `capabilities/default.json`
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "Додати явні localhost:11434 записи в Tauri capabilities", because wildcard-патерн `http://**` не матчить URL із явним портом у Tauri urlpattern-парсері. `https://`-запити (supadata, page-meta) працювали, бо HTTPS дефолтний порт прихований. Довелось додати явні `http://localhost:11434/*` і `http://127.0.0.1:11434/*`.

### Consequences
* Good, because transcript фіксує очікувану користь: після зміни capabilities і перезапуску dev-застосунку кнопка «Перекласти» запрацювала.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
- Файл: `app/src-tauri/capabilities/default.json`, розділ `permissions → http:default → allow`
- Зміни підхоплюються лише після перезапуску (`bun run start`) — capabilities вшиваються в Rust-контекст під час збірки
- Ollama піднятий через `nohup ollama serve >/tmp/ollama.log 2>&1 &`

## ADR Підключення Ollama як провайдера до pi.dev

## Context and Problem Statement
Потрібно було підключити локальні Ollama-моделі до pi coding-agent для порівняльного тестування агентних скілів (`/n-fix`). pi не має вбудованого `ollama`-провайдера, а передача `OPENAI_BASE_URL` через env не спрацювала (агент ігнорував змінну і йшов на офіційний OpenAI endpoint).

## Considered Options
* Кастомний провайдер через `~/.pi/agent/models.json` (офіційний механізм pi)
* Передача `OPENAI_BASE_URL` через env
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "`~/.pi/agent/models.json` з `api: openai-completions`", because pi підтримує кастомні провайдери через цей файл (задокументовано в `$PKG/docs/models.json`); Ollama надає OpenAI-сумісний `/v1` endpoint. Передача через env ігнорувалась агентом.

### Consequences
* Good, because transcript фіксує очікувану користь: `pi --list-models` побачив обидві Ollama-моделі; базовий tool-calling (`bash echo TOOLS_OK`) спрацював для `qwen2.5-coder:7b`.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
- Файл: `~/.pi/agent/models.json` з полями `baseUrl`, `api: "openai-completions"`, `apiKey: "ollama"`, `compat.supportsDeveloperRole: false`, `compat.supportsReasoningEffort: false`
- Поле `cost` вимагає всі чотири підполя: `input`, `output`, `cacheRead`, `cacheWrite` — без них `pi --list-models` видає `Invalid models.json schema`
- Ollama запущений з контекстом 32k: `OLLAMA_NUM_CTX=32768 nohup ollama serve`
- Документація провайдерів: `$PKG/docs/models.md` (де `$PKG=/opt/homebrew/Cellar/pi-coding-agent/0.78.0/libexec/lib/node_modules/@earendil-works/pi-coding-agent`)

## ADR Придатність локальних Ollama-моделей для агентного /n-fix через pi

## Context and Problem Statement
Проведено порівняльний тест 6 локальних Ollama-моделей (qwen2.5-coder:7b, granite3.3:8b, llama3.1:8b, qwen3:8b, deepseek-coder-v2:16b, gemma4:e4b) на здатність виконати агентний скіл `/n-fix` через pi на Apple M5 з 16 GB RAM. Тест: фіксований набір порушень (заборонений `yarn.lock`, `prettier` у devDeps, відсутній скрипт `lint-image`) → `pi --skill /n-fix` → реверифікація `npx @nitra/cursor fix`.

## Considered Options
* Локальні Ollama-моделі (різні розміри та родини)
* Хмарні моделі через pi (openai-codex/gpt-5.x, Anthropic) — не тестувались, але визначені як рекомендована альтернатива
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "Хмарні моделі для агентних скілів; локальні — тільки для генерації тексту", because жодна з 6 локальних моделей не впоралась із `/n-fix`: qwen2.5-coder:7b і granite3.3:8b емітують tool-виклики текстом (pi їх не виконує), deepseek-coder-v2:16b не підтримує tools, llama3.1:8b і gemma4:e4b викликають tools з помилками і деструктивно (gemma4 збільшила кількість ❌ з 10 до 14), qwen3:8b частково пофіксила (10→7), але не вклалась у 10 хв і створила нове порушення.

### Consequences
* Good, because transcript фіксує очікувану користь: висновок дозволяє не витрачати час на налаштування локальних агентів і чітко поділяє use-case: переклад (gemma3:4b ✅) vs агентне кодування (хмарні).
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
- Тестовий стенд: `rsync`-копія репо + `ln -s node_modules` (APFS hardlink), інжектор `/tmp/inject_violations.sh`, раннер `/tmp/nfix_run2.sh`, watchdog 600 с
- Worktree-preflight скіла обходиться розміщенням копії за шляхом `.worktrees/run-fix/` — тоді `git rev-parse --show-toplevel` містить `.worktrees` і preflight вважає умову виконаною
- RAM-footprint при `num_ctx 32768`: qwen2.5-coder:7b 8.2G, granite3.3:8b 13.9G, llama3.1:8b 11.2G, qwen3:8b 10.2G, gemma4:e4b 11.1G
- Tool-calling через Ollama OpenAI-compat (`/v1`) ненадійний: qwen2.5-coder і granite емітують `{"name":"bash",...}` як markdown-текст, а не структурований `tool_call` — pi не виконує такі виклики
