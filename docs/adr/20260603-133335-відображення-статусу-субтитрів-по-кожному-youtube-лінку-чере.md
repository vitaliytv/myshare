## ADR Відображення статусу субтитрів по кожному YouTube-лінку через `availableLangs` supadata API

## Context and Problem Statement
У застосунку `myshare` кнопка «Субтитри» з'являлась по кожному YouTube-лінку, але реальна наявність субтитрів (українські, англійські, жодних) була невідома до тапу. Потрібно було показувати статус одразу, не тягнучи весь транскрипт і не палячи квоту supadata (free tier — 100 запитів/місяць).

## Considered Options
* Один запит до transcript-endpoint supadata (без `lang`), зчитати поле `availableLangs` із відповіді — список усіх доступних мов.
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "Один запит до supadata `/youtube/transcript` без параметра `lang`, поле `availableLangs` у відповіді", because supadata повертає `availableLangs` у будь-якій відповіді transcript-endpoint, тож одного запиту достатньо щоб дізнатись усі доступні мови без завантаження повного тексту.

### Consequences
* Good, because transcript фіксує очікувану користь: один мережевий запит на відео замість двох (uk→404→en) при кожному відкритті.
* Good, because список мов кешується в `localStorage` за `videoId` (`app/src/caption-langs.js`, `STORAGE_KEY`) — повторні відкриття того самого відео не витрачають квоту.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
- Нова Rust-команда `yt_list_languages(video_id)` у `app/src-tauri/src/youtube.rs`, зареєстрована в `app/src-tauri/src/lib.rs`.
- JS-обгортка `getYoutubeLanguages` у `app/src/youtube.js`.
- Чиста функція `captionStatus()` у `app/src/caption-langs.js`: нормалізує `uk-UA`→`uk`, `en-US`→`en`; повертає `{ kind: 'uk'|'en'|'none', langs }`.
- UI: чіп 🇺🇦 UA / 🇬🇧 EN / «Без UA·EN» з tooltip повного переліку мов (`app/src/App.vue`).
- Відео без субтитрів (supadata повертає 404) → порожній список `availableLangs`.

## ADR Переклад EN→UA через локальний Ollama напряму з JS без нової Rust-команди

## Context and Problem Statement
Коли по YouTube-лінку є тільки англійські субтитри (статус `en`), потрібно перекласти їх на українську і зберегти результат. Ollama запущено локально на `localhost:11434`; застосунок — Tauri (desktop).

## Considered Options
* Новий Rust Tauri-command для HTTP-запитів до Ollama.
* HTTP-запит напряму з JS через `tauri-plugin-http` (як у `page-meta.js`).
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "HTTP-запит до Ollama з JS через `tauri-plugin-http`", because `tauri-plugin-http` вже використовується в `page-meta.js` для зовнішніх запитів і дозволяє звернутись до `localhost:11434` без додаткового Rust-коду.

### Consequences
* Good, because transcript фіксує очікувану користь: нова функціональність реалізована без нової Rust-команди та компіляції `src-tauri`.
* Good, because функція доступна тільки на desktop (`canTranslate = !isAndroidPlatform()`), бо Ollama — локальний сервіс.
* Bad, because для роботи потрібно явно додати `http://localhost:11434/*` і `http://127.0.0.1:11434/*` до `app/src-tauri/capabilities/default.json` — wildcard `http://**` не матчить URL із нестандартним портом у Tauri.

## More Information
- `app/src/ollama.js`: `chunkText` (ріже текст ≤3500 символів по межах абзаців), `translateChunk` → `POST /api/chat` (`stream:false`, `temperature:0.2`, `num_ctx:8192`), `resolveModel` → `GET /api/tags`, `translateToUkrainian({onProgress})`.
- `app/src/translation-cache.js`: кеш перекладів у `localStorage` за `videoId`; відкидає биті записи через `isValidEntry`.
- `app/src-tauri/capabilities/default.json`: додано `http://localhost:11434/*` і `http://127.0.0.1:11434/*`.
- Переклад відображається в діалозі `.cmp-grid` (двоколонка: оригінал EN | переклад UA, вирівняні посегментно).

## ADR Дефолт-модель Ollama `gemma3:4b` замість `gemma4:e4b` для перекладу субтитрів

## Context and Problem Statement
Перший запуск перекладу через `gemma4:e4b` (8.9 GB у RAM) спричинив критичне гальмування всього комп'ютера: `swap used = 16 GB`, `Pageouts: 1.6M`, 100% пам'яті під тиском. Система — Apple M5, 16 GB unified memory.

## Considered Options
* Залишити `gemma4:e4b` (8.9 GB у RAM) як дефолт.
* Перейти на `gemma3:4b` (~3.3 GB на диску, 4.4 GB у RAM при завантаженні).
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "`gemma3:4b`", because на 16 GB RAM `gemma4:e4b` (8.9 GB ваг) не влазить без свопу разом із запущеними IDE, браузером та build-процесами; `gemma3:4b` займає 4.4 GB у RAM, повністю йде на GPU (`100% GPU` за `ollama ps`), і дає адекватну українську мову для субтитрів.

### Consequences
* Good, because transcript фіксує очікувану користь: після переходу `swap used` впав з 16 GB до 3.8 GB; модель завантажується за ~1.3 с проти ~3.2 с у `gemma3:4b` (виміряно в bench).
* Good, because `resolveModel` у `ollama.js` відкочується до першої наявної моделі, якщо `DEFAULT_MODEL` не встановлена — нова константа лише змінює пріоритет.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
- Константа `DEFAULT_MODEL = 'gemma3:4b'` у `app/src/ollama.js`.
- Живий тест перекладу `gemma3:4b` підтверджений вручну через `curl POST /api/chat`: "Never gonna give you up…" → "Ніколи не здамся…" (адекватна українська, переноси збережено).
- `ollama list` підтверджує установку: `gemma3:4b a2af6cc3eb7f 3.3 GB`.

## ADR `keep_alive: '5m'` і `num_ctx: 8192` для Ollama-запитів перекладу

## Context and Problem Statement
Під час розробки ollama-перекладу спочатку було встановлено `keep_alive: 0` (негайне вивантаження після перекладу) для звільнення RAM. Але повторні переклади й багаточанкові тексти потребували перевантаження моделі на кожен чанк. Також не було обмежено KV-кеш (`num_ctx`), що роздувало споживання пам'яті.

## Considered Options
* `keep_alive: 0` (негайне вивантаження після кожного чанку/перекладу).
* `keep_alive: '5m'` (модель тримається 5 хвилин після останнього запиту).
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "`keep_alive: '5m'` з `num_ctx: 8192`", because користувач явно попросив повернути 5-хвилинне утримання — модель не перевантажується між чанками одного перекладу й при повторних перекладах упродовж кількох хвилин; `num_ctx: 8192` обмежує KV-кеш до реального розміру чанку (~1200 токенів).

### Consequences
* Good, because transcript фіксує очікувану користь: повторний переклад того ж відео протягом 5 хв не вимагає перезавантаження моделі.
* Good, because `num_ctx: 8192` підтверджений у виводі `ollama ps` — параметр застосувався.
* Bad, because `keep_alive: '5m'` утримує ~4.4 GB (gemma3:4b) у RAM протягом 5 хвилин після перекладу, навіть якщо інших перекладів не буде.

## More Information
- `app/src/ollama.js`, функція `translateChunk`: параметр `options: { temperature: 0.2, num_ctx: 8192 }`, `keep_alive: '5m'` у тілі запиту.
- `unloadModel` (функція з `keep_alive: 0`) введена, а потім видалена за рішенням користувача.
- Підтверджено в `ollama ps`: `CONTEXT 8192`, `UNTIL 4 min`.
