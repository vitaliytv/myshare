## ADR Визначення наявності субтитрів через поле `availableLangs` з одного запиту

## Context and Problem Statement
Застосунок myshare відображає список YouTube-URL і мав показувати статус наявності субтитрів по кожному. Питання: робити окремі запити на кожну мову чи скористатися тим, що supadata повертає вичерпний список усіх мов субтитрів в одній відповіді?

## Considered Options
* Окремі запити `?lang=uk` / `?lang=en` (два HTTP-запити на відео)
* Один запит без `lang` → supadata повертає `availableLangs` у будь-якій відповіді

## Decision Outcome
Chosen option: "Один запит без `lang` → поле `availableLangs`", because supadata за власної документацією включає `availableLangs` в усі відповіді transcript-endpoint, що дає вичерпний список мов за одне звернення й вдвічі знижує витрату API-квоти (free tier — 100 запитів/місяць).

### Consequences
* Good, because transcript фіксує очікувану користь: один запит на відео замість двох, квота витрачається вдвічі повільніше.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Нова Tauri-команда `yt_list_languages(video_id)` у `app/src-tauri/src/youtube.rs`; JS-обгортка `getYoutubeLanguages` у `app/src/youtube.js`; чиста функція `captionStatus` у `app/src/caption-langs.js`. Endpoint: `GET /v1/youtube/transcript?videoId=…&text=true` (без параметра `lang`).

## ADR Кешування списків мов субтитрів і перекладів у `localStorage`

## Context and Problem Statement
Кожне звернення до supadata (`availableLangs`) і кожен переклад через Ollama коштують ресурси (API-квота і CPU/GPU відповідно). При повторному відкритті того самого відео без кешу обидва виклики відбуваються заново.

## Considered Options
* Кешування у `localStorage` за ключем `videoId`
* Без кешу — кожного разу новий запит

## Decision Outcome
Chosen option: "Кешування у `localStorage` за ключем `videoId`", because supadata free tier — 100 запитів/місяць, а Ollama-переклад займає від 14 до 40 секунд; повторний перегляд того самого відео має бути миттєвим.

### Consequences
* Good, because transcript фіксує очікувану користь: повторні відкриття відео не витрачають API-квоту і не запускають Ollama повторно.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Модулі: `app/src/caption-langs.js` (функції `loadLangsCache`, `saveLangsCache`, константа `STORAGE_KEY`) та `app/src/translation-cache.js` (функції `loadTranslations`, `saveTranslations`, `isValidEntry`). Кеш прив'язується до `videoId`, а не до URL.

## ADR Переклад субтитрів EN→UA через локальний Ollama (desktop-only)

## Context and Problem Statement
Застосунок показував статус «тільки EN субтитри» для частини YouTube-відео. Потрібно дати користувачу можливість отримати переклад українською без залучення хмарних платних API.

## Considered Options
* Локальний Ollama (безкоштовно, privacy-friendly, desktop-only)
* Хмарний перекладний API (платний, потребує мережі)

## Decision Outcome
Chosen option: "Локальний Ollama", because він безкоштовний, не надсилає дані назовні і вже встановлений на Mac розробника; функція захищена умовою `!isAndroidPlatform()`, бо на Android Ollama відсутній.

### Consequences
* Good, because transcript фіксує очікувану користь: переклад коштує $0, текст субтитрів не покидає пристрій.
* Bad, because transcript фіксує: на 16 GB RAM великі моделі (≥8B) спричиняють swap-трешинг; обмежено desktop-платформою.

## More Information
Реалізовано в `app/src/ollama.js`: `POST http://localhost:11434/api/chat` (`stream: false`, `temperature: 0.2`, `num_ctx: 8192`, `keep_alive: '5m'`). Кнопка «Перекласти» видима тільки коли `captionStatus.kind === 'en'` та `!isAndroidPlatform()`.

## ADR HTTP-запити до Ollama безпосередньо з JS через `tauri-plugin-http`

## Context and Problem Statement
Застосунок використовує Tauri. Для звернення до локального HTTP-сервісу (Ollama) можна або написати нову Rust Tauri-команду, або використовувати наявний `tauri-plugin-http` прямо з JS (як вже робить `page-meta.js`).

## Considered Options
* Нова Rust Tauri-команда `yt_translate` (проксі до Ollama)
* Прямий виклик `tauri-plugin-http` з JS (як у `page-meta.js`)

## Decision Outcome
Chosen option: "Прямий виклик `tauri-plugin-http` з JS", because Ollama слухає localhost — це вже дозволений transport; додавати Rust-прошарок без потреби суперечить принципу «не вводити абстракцій без вимоги».

### Consequences
* Good, because transcript фіксує очікувану користь: нуль змін у Rust, менший обсяг коду.
* Bad, because transcript фіксує: Tauri HTTP scope `http://**` не матчить URL з явним портом `:11434`; довелося додати явний дозвіл `http://localhost:11434/*` у `app/src-tauri/capabilities/default.json`.

## More Information
Дозвіл додано до `app/src-tauri/capabilities/default.json` у секції `http:default → allow`. Без цього fix requests провалювались із `Network Error` всередині WebView.

## ADR Вибір дефолтної моделі `gemma3:4b` для перекладу субтитрів

## Context and Problem Statement
Початково використовувалася `gemma4:e4b` (9.6 GB на диску, ~10.6 GB у RAM). На Mac з 16 GB unified RAM це спричиняло swap-трешинг (swap used = 16 GB) і гальмувало весь комп'ютер. Потрібна модель, що влазить у RAM без свопу та дає прийнятну якість перекладу українською.

## Considered Options
* `gemma4:e4b` (9.6 GB / ~10.6 GB RAM)
* `gemma3:4b` (3.3 GB / ~4.4 GB RAM)
* `aya-expanse:8b`, `aya:8b`, `command-r7b` (4.8–5.1 GB / ~6.4–6.8 GB RAM) — мультимовні спеціалізовані

## Decision Outcome
Chosen option: "`gemma3:4b`", because бенчмарк показав: 4.4 GB RAM (≈ 2.4× менше за e4b), 37.5 tok/s, повний текст за ~14 s, якість перекладу друга після e4b і вища за всіх спеціалізованих мультимовних (aya/command-r7b давали граматичні помилки й калькування). На 16 GB це лишає ~11 GB headroom без свопу.

### Consequences
* Good, because transcript фіксує очікувану користь: без swap-трешингу навіть з відкритими IDE; швидший load (1.7 s vs 6.3 s у e4b).
* Bad, because якість дещо нижча за `gemma4:e4b` (незначні помилки особи дієслова); `gemma3:4b` не підтримує tool-calling (Ollama повертає `does not support tools`), тому не придатна для агентних задач через pi CLI.

## More Information
Константа `DEFAULT_MODEL = 'gemma3:4b'` у `app/src/ollama.js`. Бенчмарк проводився на субтитрах відео `dQw4w9WgXcQ` (EN, 2089 символів). Метрики RAM отримано через `GET http://localhost:11434/api/ps` після кожного inference-виклику.

## ADR `keep_alive: '5m'` для Ollama-моделі під час перекладу

## Context and Problem Statement
Переклад виконується посегментно (chunking). Між чанками модель треба тримати завантаженою або вивантажувати. Спочатку для звільнення RAM після перекладу використовувалося негайне вивантаження (`keep_alive: 0`), але це знову і знову перевантажує модель при повторних запитах у рамках одного перекладу.

## Considered Options
* `keep_alive: 0` — вивантаження одразу після кожного чанку
* `keep_alive: '5m'` — стандартний Ollama timeout (модель лишається 5 хвилин)

## Decision Outcome
Chosen option: "`keep_alive: '5m'`", because 5-хвилинне вікно покриває весь час перекладу (одного відео вистачає ~14–40 s), запобігає повторному завантаженню моделі між чанками і при повторному зверненні протягом сесії. Обрано за прямою вимогою користувача після того, як `keep_alive: 0` довів себе зайво агресивним.

### Consequences
* Good, because transcript фіксує очікувану користь: модель не перевантажується між чанками; повторний переклад в межах 5 хвилин не витрачає час на load (~1.7 s на завантаження).
* Bad, because після завершення перекладу ~4.4 GB лишаються в RAM до 5 хвилин; на 16 GB це прийнятно, але не оптимально для паралельної роботи.

## More Information
`app/src/ollama.js`, параметр `keep_alive` у тілі `POST /api/chat`. `num_ctx: 8192` лишається в силі — обмежує KV-кеш незалежно від keep_alive.

## ADR Підключення локального Ollama до pi CLI через `~/.pi/agent/models.json`

## Context and Problem Statement
pi CLI (`pi-coding-agent 0.78.0`) підтримує кастомні провайдери. Для тестування coding-моделей через агентний скіл n-fix потрібно підключити локальний Ollama до pi без модифікації коду pi.

## Considered Options
* Змінні середовища `OPENAI_BASE_URL` + `OPENAI_API_KEY=ollama` при запуску pi
* Конфіг `~/.pi/agent/models.json` з провайдером `ollama` (задокументований спосіб)

## Decision Outcome
Chosen option: "`~/.pi/agent/models.json`", because змінна `OPENAI_BASE_URL` ігнорується pi (при спробі він звернувся до справжнього api.openai.com і отримав 401); офіційний спосіб — файл `models.json` з полями `baseUrl`, `api: 'openai-completions'`, `apiKey: 'ollama'`.

### Consequences
* Good, because transcript фіксує очікувану користь: `pi --list-models ollama` показав обидві моделі (`gemma3:4b`, `qwen2.5-coder:7b`); `qwen2.5-coder:7b` успішно виконала bash tool-call через pi.
* Bad, because transcript фіксує: `gemma3:4b` повертає `does not support tools` (Ollama-збірка gemma3 без function-calling), тому для агентного n-fix через pi придатна тільки `qwen2.5-coder:7b`.

## More Information
Файл: `~/.pi/agent/models.json`. Схема `cost` вимагає чотири поля (`input`, `output`, `cacheRead`, `cacheWrite`) — без `cacheRead`/`cacheWrite` pi відмовляється завантажувати конфіг з помилкою `Invalid models.json schema`. Документація: `$PI_PKG/docs/models.md` (custom-provider.md).
